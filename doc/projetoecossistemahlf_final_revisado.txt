## 1. Visão Geral e Propósito

O Ecossistema HLF Pro2Col é um conjunto de páginas web, scripts e ativos
digitais projetado para:

- Atrair e qualificar prospects por meio de conteúdo consultivo e geolocalizado
- Direcionar leads para conversão via WhatsApp, catálogo oficial e cadastro Herbalife
- Sustentar autoridade local em Novo Hamburgo/RS e região do Vale do Sinos
- Operar com performance, conformidade e rastreabilidade

### 1.1 Princípios Fundamentais

- Atendimento humano assistido: respostas rápidas com apoio de IA consultiva
  e atalhos de digitação — sem bots conversacionais
- Identidade visível: toda página exibe, acima da dobra, a identificação
  "Distribuidor Independente Herbalife: Edu Sidegum"
- Uso padrão de componentes compartilhados: CSS, scripts, header, footer e
  blocos legais são mantidos em diretório comum
- Conformidade total: disclaimer oficial Herbalife presente em todas as páginas
- SEO local: Novo Hamburgo/RS como âncora geográfica principal

### 2.4 Agregador de Links (links.html)

O arquivo links.html na raiz do projeto funciona como hub centralizador de
todos os links do ecossistema. Sua função é:

- Ser a única URL divulgada nos scripts de atendimento (atalhos de teclado)
- Evitar links quebrados quando uma página mudar de endereço
- Permitir atualização centralizada sem editar scripts individuais
- Servir como ponto único de entrada para leads em diferentes estágios do funil

Regra: Atualizar o links.html sempre que uma nova página for criada ou um
link for alterado.

### 2.5 Composição Visual de Cada Página

1. Header/Topbar — fixa, identidade + navegação
2. Header-spacer — compensa altura da topbar fixa
3. Hero acima da dobra — H1 local, subtítulo, CTA para WhatsApp,
   identificação visível do distribuidor
4. Body — dores/problemas, tabelas nutricionais, listas do protocolo,
   conexão local
5. Footer — semântico, cross-linking, disclaimer, links oficiais

## 3. Head Template Padrão

O head_padrao.txt do diretório menu/templates/ é o template mestre.
Copia-se integralmente para cada página, alterando apenas os campos
variáveis:

- <title>
- <meta name="description">
- <meta property="og:title">
- <meta property="og:description">
- <link rel="canonical"> href
- href do preload da imagem LCP
- Campos variáveis do JSON-LD

ATENÇÃO: O caminho do bloco-incluir.js aponta para assets/js/ dentro do
próprio projeto (não para um diretório externo).

### 3.3 Regras

- Uso padrão do template para todas as páginas
- Elementos marcados como FIXOS não podem ser removidos
- Elementos marcados como VARIÁVEIS são alterados por página
- CSS adicional específico só com justificativa técnica registrada
- O caminho do bloco-incluir.js é sempre assets/js/bloco-incluir.js
  (relativo ao projeto)
- O canonical deve sempre apontar para o arquivo real (.html explícito),
  nunca para a forma curta de diretório (ex: usar hypedrink/index.html,
  não hypedrink/)
- Links públicos (WhatsApp, topbar, footer) podem usar a forma curta
  (hypedrink/) — o GitHub Pages resolve automaticamente para index.html.
  Mas a tag <link rel="canonical"> dentro de cada página deve sempre
  apontar para o arquivo real.

## 5. Scripts JS Compartilhados

### 5.1 Tabela de Scripts

| Script | Função | Carregamento |
|---|---|---|
| bloco-incluir.js | Injeção dinâmica de topbar, footer, cookies e blocos compartilhados | defer |
| bloco-protecao.js | Proteção de conteúdo (desabilita clique direito, seleção, arrastar) | defer |
| dinamico.js | Funcionalidades dinâmicas (scroll suave, animações de entrada, tabs) | defer |
| novo_dir_dinamico.js | Navegação avançada e criação de diretórios dinâmicos | defer |
| bloco-proposta.js | Monta e apresenta a proposta de kit de produtos na pasta /perfil/ | defer |

### 5.2 Regras de Uso

- Todos os scripts residem em assets/js/ exceto bloco-proposta.js que reside
  em scripts/ (fora de assets/js/, conforme estrutura real)
- Todos os scripts são carregados com atributo defer
- O bloco-incluir.js é o único script referenciado no <head> do template
  base; os demais são injetados dinamicamente por ele ou carregados
  conforme necessidade da página
- bloco-proposta.js é carregado apenas por avaliacao.html (página de
  propostas em /perfil/)
- Scripts em /scripts/ e /assets/js/ são bloqueados para rastreamento
  via robots.txt (ver Anexo 1 §3.4) — isso não impede o carregamento
  funcional via <script src> no navegador do usuário

## 6. SEO e Dados Estruturados

### 6.1 Meta-Tags por Página

Toda página indexável deve conter as meta-tags robots conforme Anexo 1 §4.3.
Toda página não-indexável deve conter as meta-tags robots conforme Anexo 1 §4.4.
A Seção 10.1 (Estrutura Obrigatória) referencia o Anexo 1 §4.3/§4.4 como
única fonte de verdade para meta robots — não usar versão simplificada.

### 6.2 Schema JSON-LD

Todo JSON-LD deve usar @graph com referências @id, sempre incluindo /HLF/
no domínio raiz oficial:

```json
{
  "@context": "https://schema.org",
  "@graph": [
    {
      "@type": "Person",
      "@id": "https://edusidegum.github.io/HLF/#person",
      "name": "Eduardo Sidegum",
      "jobTitle": "Distribuidor Independente Herbalife"
    },
    {
      "@type": "LocalBusiness",
      "@id": "https://edusidegum.github.io/HLF/#business",
      "name": "Eduardo Sidegum | Distribuidor Independente Herbalife",
      "founder": { "@id": "https://edusidegum.github.io/HLF/#person" },
      "address": {
        "@type": "PostalAddress",
        "addressLocality": "Novo Hamburgo",
        "addressRegion": "RS",
        "addressCountry": "BR"
      }
    },
    {
      "@type": "WebSite",
      "@id": "https://edusidegum.github.io/HLF/#website",
      "url": "https://edusidegum.github.io/HLF/",
      "publisher": { "@id": "https://edusidegum.github.io/HLF/#person" }
    },
    {
      "@type": "BreadcrumbList",
      "itemListElement": [
        {
          "@type": "ListItem",
          "position": 1,
          "name": "Home",
          "item": "https://edusidegum.github.io/HLF/"
        },
        {
          "@type": "ListItem",
          "position": 2,
          "name": "[NOME_DA_PAGINA]",
          "item": "[URL_CANÔNICA]"
        }
      ]
    }
  ]
}
```

### 6.3 Tipos de JSON-LD por Página

| Tipo de página | JSON-LD obrigatório |
|---|---|
| Landing / Programa | WebPage + Organization + BreadcrumbList |
| Ferramenta (perfil) | WebPage + Organization |
| Cadastro | WebPage + Organization |
| Avaliação (noindex) | Nenhum (não será indexado) |

### 6.4 Hierarquia de Cabeçalhos (H1–H6)

- Um único <h1> por página — deve conter o título principal com a keyword
  do projeto
- <h2> para seções principais (Hero, Contexto, Conversão)
- <h3> para subseções dentro de cada <h2>
- <h4>–<h6> apenas quando houver profundidade real de conteúdo

Padrão por página:

```html
<h1>[Título do Projeto] | Distribuidor Herbalife</h1>
  <h2>O que é [Projeto]</h2>
    <h3>Benefícios</h3>
    <h3>Como funciona</h3>
  <h2>Seus Resultados</h2>
  <h2>Próximos Passos</h2>
```

### 1. Regra de Preservação de Conteúdo (Anti-Simplificação)

#### 1.1. Manifesto de Componentes Obrigatórios

Toda página gerada DEVE conter, nesta ordem:

1. [Header Topbar] — com logo, identificação do distribuidor e navegação
2. [Hero Section] — com H1 único e subtítulo de contexto
3. [Context Content Block] — conteúdo informativo aprovado
4. [Conversion Loop] — Trio de Saída (ver Seção 4)
5. [Footer HLF] — com disclaimer, links legais e identificação

#### 1.2. Proibição de Sumarização

É terminantemente proibido reduzir, parafrasear ou omitir blocos de texto
fornecidos como "Contexto de Página". O conteúdo deve ser transposto
integralmente, otimizando:

- Marcação HTML semântica (H1–H6, <article>, <section>, <nav>, <aside>)
- Atributos de performance (loading="lazy" em imagens não-LCP, preload,
  preconnect, fetchpriority)
- Acessibilidade (ARIA labels, alt descritivo, role, tabindex)
- Estrutura de dados estruturados (JSON-LD)

Sem jamais reduzir o conteúdo informativo aprovado.

#### 1.3. Exceção de Performance

Minificação de CSS e JS é permitida, desde que:

- O arquivo original não-minificado seja preservado no repositório
- O arquivo minificado receba sufixo .min.css ou .min.js
- A funcionalidade seja idêntica à versão original

### 2. Injeção Dinâmica via PowerShell (Rastreamento)

#### 2.1. Variáveis Obrigatórias

Os scripts de deploy devem utilizar obrigatoriamente as variáveis:

    $GA4_ID
    $GTM_ID
    $META_PIXEL_ID

#### 2.2. Tratamento de Variável Nula

Se a variável for nula, o script deve injetar um comentário HTML:

#### 2.3. Ordem de Inserção

### 3. Blindagem de Bots (Preços e Lógicas)

#### 3.1. Nível de Arquivo — Páginas HTML com Valores Explícitos

Todo arquivo HTML que exiba tabela de preços, valores de programas ou
fórmulas de decisão deve receber, no <head>:

    <meta name="robots" content="noindex, nofollow">

#### 3.2. Nível de Arquivo — Páginas Dependentes de Estado

Páginas que dependem de localStorage, sessionStorage ou interação prévia
para renderizar conteúdo (ex: avaliacao.html) devem receber:

    <meta name="robots" content="noindex, follow">

O follow permite que os bots sigam links do footer/topbar, mas não
indexem o conteúdo dinâmico.

#### 3.3. Nível de Arquivo — Scripts JS Puros

Arquivos .js que contenham lógica de decisão (ex: bloco-proposta.js,
novo_dir_dinamico.js) devem receber no topo do arquivo:

    /* HLF — Arquivo protegido: noindex, nofollow */

E ser bloqueados via robots.txt (ver Seção 3.4).

#### 3.4. Nível de Diretório — robots.txt Canônico

O robots.txt na raiz do repositório DEVE conter exatamente:

    # Robots.txt — Ecossistema HLF
    # Regra geral (todos os bots)
    User-agent: *
    Allow: /

    Disallow: /data/

    Disallow: /assets/js/novo_dir_dinamico.js

    User-agent: Googlebot
    Allow: /
    Disallow: /doc/
    Disallow: /data/
    Disallow: /scripts/
    Disallow: /assets/js/novo_dir_dinamico.js

    # Bingbot — regra específica
    User-agent: bingbot
    Allow: /
    Disallow: /doc/
    Disallow: /data/
    Disallow: /scripts/
    Disallow: /assets/js/novo_dir_dinamico.js

#### 3.5. Observação sobre GitHub Pages

O GitHub Pages NÃO suporta .htaccess nem cabeçalhos HTTP customizados

- Meta tag inline no <head> (para HTML)

### 4. Padrão de JSON-LD por Tipo de Página

| Tipo de página | JSON-LD obrigatório |
|---|---|
| Landing / Programa | WebPage + Organization + BreadcrumbList |

### 5. Hierarquia de Cabeçalhos (H1–H6)

#### 5.1. Regra Obrigatória

- Um único <h1> por página — deve conter o título principal com a keyword
  do projeto
- <h2> para seções principais (Hero, Contexto, Conversão)
- <h3> para subseções dentro de cada <h2>
- <h4>–<h6> apenas quando houver profundidade real de conteúdo

#### 5.2. Padrão por Página

```html
<h1>[Título do Projeto] | Distribuidor Herbalife</h1>
  <h2>O que é [Projeto]</h2>
    <h3>Benefícios</h3>
    <h3>Como funciona</h3>
  <h2>Seus Resultados</h2>
  <h2>Próximos Passos</h2>
```

### 7. Tratamento de Imagens

#### 7.1. Formato

- Todas as imagens devem ser .webp
- LCP image deve ter fetchpriority="high" e preload no <head>
- Imagens não-críticas devem ter loading="lazy"

#### 7.2. Atributos Obrigatórios

```html
<img src="[URL].webp"
     alt="[Descrição descritiva e específica]"
     width="[LARGURA]"
     height="[ALTURA]"
     loading="lazy"
     decoding="async">
```

#### 7.3. Imagem LCP (acima da dobra)

```html
<link rel="preload" as="image" href="https://edusidegum.github.io/HLF/img/[LCP_IMAGE].webp">
```

```html
<img src="https://edusidegum.github.io/HLF/img/[LCP_IMAGE].webp"
     alt="[Descrição]"
     width="[LARGURA]"
     height="[ALTURA]"
     fetchpriority="high"
     decoding="async">
```

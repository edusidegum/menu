Instrução Técnica: Arquitetura Universal do Ecossistema HLF (Revisada)

Ao Programador — Escopo e Intenção
Esta instrução consolida a arquitetura centralizada do Ecossistema HLF com um ajuste crítico: o MENU NÃO É universal. A inclusão de cabeçalho/menu deve ser decidida caso a caso no briefing. O rodapé (footer), os blocos de rastreamento e os documentos legais continuam centralizados por padrão, salvo instrução em contrário.

Premissas Globais
- Diretriz global não fixa paths de repositório; [BASE_URL] e [BASE_PATH] são definidos no briefing do projeto.
- Não duplicar código; preferir componentes centralizados quando aplicável e autorizado.
- Proteções e rastreamento seguem o repositório central (edusidegum/menu), exceto quando explicitamente desativados.

1) Integração de Scripts de Sistema (Core)
Importar, via URL pública do repositório central, os scripts base:
- Gestão de Blocos (includes seletivos): https://raw.githubusercontent.com/edusidegum/menu/main/scripts/bloco-incluir.js
- Segurança (proteção anticópia seletiva): https://raw.githubusercontent.com/edusidegum/menu/main/scripts/bloco-protecao.js

Observações de uso
- A proteção deve ser aplicada de forma seletiva, sem prejudicar acessibilidade/UX.
- O bloco-incluir é o orquestrador de componentes, mas sua aplicação ao HEADER/MENU depende do briefing (ver Seção 3).

2) Componentes Compartilhados (Include Modules)
Injetar conforme necessidade, sempre que centralização for desejada e aprovada:
Tracking
- Cookies/Consentimento: https://raw.githubusercontent.com/edusidegum/menu/main/compartilhado/bloco-cookies.html
- GA4/GTM (ou outro, se substituído globalmente): https://raw.githubusercontent.com/edusidegum/menu/main/compartilhado/bloco-ga4-gtm.html

Legal e Compliance (padrão)
- Política de Privacidade: https://raw.githubusercontent.com/edusidegum/menu/main/compartilhado/politicadeprivacidade.html
- Termos de Uso e Identificação (Eduardo Sidegum): https://raw.githubusercontent.com/edusidegum/menu/main/compartilhado/Termos_de_Uso_e_Identificacao_Eduardo_Sidegum.html

3) Política de Cabeçalho/Menu e Rodapé (DECISÃO DE BRIEFING)
O menu não é universal. Antes de qualquer inclusão automática, RESPONDA às perguntas abaixo:
Q1. O MENU é compartilhado (padrão do ecossistema) ou específico do projeto?
- Se COMPARTILHADO: o bloco-incluir pode injetar o menu padrão central.
- Se ESPECÍFICO: NÃO injetar menu central. O projeto deve fornecer seu próprio header/menu local. O bloco-incluir pode ser usado apenas para o footer.

Q2. O FOOTER será compartilhado (central) ou específico?
- Recomendação: usar footer central sempre que possível para manter rastreamento/legal/autoridade consistentes.
- Alternativa: criar um include dedicado somente ao footer (ex.: bloco-footer.html) caso o menu seja específico.

Q3. O projeto é Pessoal (do Eduardo) ou de Terceiros (cliente/parceiro)?
- Pessoal: documentos legais e autoridade do Eduardo; rodapé deve indicar “Desenvolvido em dd/mm/aaaa”.
- Terceiros: manter rastreamento/legais conforme contrato; a referência de autoridade difere (ver Seção 4).

4) Estratégia de SEO, Autoridade e Rodapé por Tipo de Projeto
Autoridade e Links de Relevância
- Projeto Pessoal (Eduardo): incluir link de autoridade para https://edusidegum.github.io/edusidegum/sobre.html
- Projeto de Terceiros: apontar para https://edusidegum.github.io/edusidegum/index.html (referência institucional)

Rodapé — Textos e Metainformação
- Projeto Pessoal: exibir “Desenvolvido em dd/mm/aaaa” (data real de publicação) no footer.
- Projeto de Terceiros: exibir identificação conforme contrato; manter créditos técnicos quando permitido.

Legais (padrão)
- Por padrão, usar os legais centralizados do Eduardo Sidegum em projetos pessoais.
- Em projetos de terceiros, legais podem variar; se necessário, criar include legal específico do cliente e não usar o central.

5) Regras de Estilização e Assets
- Modularidade: CSS, fontes e imagens são LOCAIS por padrão.
- Exceção: compartilhamento de assets visuais entre subprojetos somente mediante instrução explícita do Eduardo.
- Controle de Acesso: classificar conteúdo entre público e restrito; aplicar proteção (bloco-protecao.js) somente onde autorizado.

6) Fluxo Operacional do Desenvolvedor (Passo a Passo)
1) Briefing obrigatório — registrar:
   - [BASE_URL] e [BASE_PATH]
   - MENU: Compartilhado ou Específico?
   - FOOTER: Compartilhado ou Específico?
   - TIPO DE PROJETO: Pessoal (Eduardo) ou Terceiros?
   - Legais: Central (Eduardo) ou Específicos do cliente?
2) Incluir scripts core (bloco-incluir.js e bloco-protecao.js)
3) Incluir módulos de tracking e legais conforme decisões do briefing
4) Se MENU compartilhado: habilitar include de header/menu central
5) Se MENU específico: não incluir menu central; habilitar apenas footer central (ou footer específico, se decidido)
6) Inserir link de autoridade correto (Seção 4) e texto de “Desenvolvido em …” quando projeto pessoal
7) Publicar, revisar acessibilidade e SEO técnico (checklist abaixo)

7) Checklist de Implementação Rápida (Padrão Ouro)
- Briefing respondido (Q1–Q3) e documentado
- [BASE_URL]/[BASE_PATH] definidos e aplicados
- Scripts core importados (includes + proteção) sem quebrar UX/A11y
- Tracking e legais corretos para o tipo de projeto
- Header/Menu conforme decisão (central ou específico)
- Footer ativo (central recomendado) com link de autoridade adequado
- Texto “Desenvolvido em dd/mm/aaaa” presente em projetos pessoais
- Caminhos absolutos/relativos coerentes com a estratégia do projeto
- Medições básicas CWV/A11y sem regressão

8) Notas de Risco e Continuidade
- Ponto Único de Falha (centralização): se o repositório central ficar indisponível, menu/footer/legais/trackers centralizados podem não carregar.
- Mitigação: manter backup do repositório central e plano de contingência (cópia local temporária quando necessário, com data de expiração).

9) Governança e Mudanças Globais
- Qualquer troca de rastreador (ex.: GA4 → outro) será feita no include central (bloco-ga4-gtm.html) e propagada automaticamente aos projetos que o utilizam.
- Toda inclusão/remoção de componentes centrais deve ser acompanhada de nota de versão e checklist de impacto (menu, footer, legais, trackers, SEO).

Encerramento
Esta é a instrução definitiva ao programador para integração universal do Ecossistema HLF, com decisão explícita de briefing sobre o MENU e a possibilidade de usar o bloco-incluir exclusivamente para o footer quando necessário. O objetivo é garantir segurança, rastreabilidade e identidade consistentes — sem impor um cabeçalho único onde ele não se aplica.


Arquitetura de Sistemas Web — Vs4
Diretriz Global (Padrão Ouro) para projetos Web e Aplicativos — com anexos operacionais
Esta versão substitui referências a repositórios específicos por variáveis de projeto. Os templates operacionais ficam nos anexos ao final, para uso direto pelo implementador.
---
1) Propósito e Escopo Global
Objetivo: padronizar construção, revisão e publicação de páginas e apps estáticos/dinâmicos, garantindo SEO técnico, performance, acessibilidade, segurança e consistência de marca.
Abrangência: sites estáticos (GitHub Pages, S3, etc.) e dinâmicos (VPS/PaaS) com as mesmas regras de markup, SEO e UX.
Princípios: conteúdo íntegro (sem sumarização indevida), componentes obrigatórios, paths consistentes, controle de bots por tipo de página, “Trio de Saída” padronizado.
2) Briefing mínimo obrigatório (antes da primeira linha de código)
O implementador deve registrar e confirmar:
[BASE_URL]: domínio/host oficial (ex.: https://www.dominio.com/ ou https://sub.dominio.com/)
[BASE_PATH]: pasta base quando aplicável (ex.: /, /HLF/, /projeto/). Se não houver, usar “/”.
Público-alvo e proposta de valor
Objetivo primário de conversão: WhatsApp, Cadastro, Venda
Abrangência regional (cidades/UF) para SEO local
Stack: Estático (Pages/CDN) ou Dinâmico (VPS/API)
Identidade: paleta, fontes locais, tema CSS (style_green.css | style_white.css | style_black.css), emojis de affordance
Observação: todas as referências absolutas nos templates devem usar [BASE_URL][BASE_PATH] como prefixo.
3) Arquitetura de Ativos e Performance
Fontes Locais: hospedar .woff2 em assets/fonts/ e declarar em CSS dedicado (fonts.css)
Ordem de Carga (head):
<link rel="preload" as="font" href="[BASE_PATH]assets/fonts/Inter-Variable.woff2" type="font/woff2" crossorigin>
<link rel="stylesheet" href="[BASE_PATH]assets/css/fonts.css">
<link rel="stylesheet" href="[BASE_PATH]assets/css/style_[TEMA].css">
Imagens: WebP obrigatório; LCP com fetchpriority="high"; secundárias com loading="lazy" e decoding="async"; width e height sempre definidos.
Componentização: bloco-includes.js com caminhos absolutos baseados em [BASE_URL][BASE_PATH]. Evitar duplicação manual de header/footer/meta.
4) SEO Técnico e Governança de Bots
Matriz de meta-robots por tipo de página:
Públicas (indexáveis): index, follow, max-snippet:-1, max-image-preview:large, max-video-preview:-1
Dependentes de estado (ex.: /perfil/avaliacao.html): noindex, follow
Com valores explícitos (tabelas de preço): noindex, nofollow, noimageindex
Scripts/dados (JS/JSON/CSV): bloquear via robots.txt (ver Anexo A)
Canonical ≡ og:url: idênticos, absolutos, com path e arquivo .html explícitos (ex.: [BASE_URL][BASE_PATH][SECAO]/[ARQUIVO].html)
Hreflang: aplicar pt-BR + x-default nas páginas indexáveis, apontando para a canônica
Estruturados (JSON-LD): usar @graph com @id estáveis (#website, #business, #webpage). Preferir referenciar entidades por @id em páginas internas
Advanced SEO: aplicar SearchAction (somente na Home), ImageObject para 1–2 imagens-chave (hero/LCP), FAQPage apenas quando houver seção visível — ver Anexo F
5) Segurança e Compliance
Proteção de Interface: bloqueios seletivos (contextmenu/Ctrl+U/S/I) apenas quando estritamente necessário; reverter se degradar INP ou acessibilidade. Nunca bloquear seleção em formulários ou atalhos assistivos.
Segregação de Lógica: manter lógica sensível fora do front-end público; quando necessário, usar APIs externas com autenticação, CORS controlado e sem expor chaves/tokens.
LGPD/Privacidade: banner de cookies somente quando rastreadores/embeds gerarem cookies; sempre manter Política de Privacidade acessível.
6) UX, Acessibilidade e Conteúdo
Conteúdo íntegro: proibido reduzir/omitir conteúdo editorial ao “adaptar” páginas; preservar contexto, benefícios, instruções e CTAs
Acessibilidade: contrastes AA+, rótulos/alt significativos, foco visível, sem bloqueios que prejudiquem leitores de tela
Navegação: header e footer consistentes; “Trio de Saída” sempre presente e funcional
WhatsApp: parâmetro padrão ?text=Vim_da_Pagina_[SLUG] (definir [SLUG] por página)
7) Governança, Versionamento e Revisão
Padrão de pastas mínimo:
/assets/css/ (fonts.css, style_green.css, style_white.css, style_black.css)
/assets/fonts/ (.woff2 locais)
/img/ (.webp)
/scripts/ (bloco-includes.js e lógicas não indexáveis)
/data/ (catálogos, JSON) — ver robots.txt
/doc/ (documentação)
Code Review: checklist obrigatório (ver Seção 9)
Publicação: validar Rich Results (FAQ/ImageObject/SearchAction), URL Inspection, Mobile-Friendly e CWV (LCP/INP/CLS ≥ 90)
8) Templates Incorporados ao Projeto (anexos)
Os anexos desta Vs4 fornecem blocos prontos e parametrizados por [BASE_URL] e [BASE_PATH]. Devem ser incluídos por include ou build.
9) Checklist de Publicação (Padrão Ouro)
Canonical absoluto ≡ og:url, com [BASE_PATH] e .html
Meta-robots conforme tipo de página (matriz da Seção 4)
Hreflang pt-BR + x-default nas indexáveis
JSON-LD válido (Rich Results Test) e @id consistentes (#website, #business)
robots.txt publicado (Anexo A) com Disallow de /doc/, /data/, /scripts/ (prefixados por [BASE_PATH])
Trio de Saída presente (Cadastro, Catálogo, WhatsApp ?text=Vim_da_Pagina_[SLUG])
Imagem LCP com fetchpriority="high"; demais com lazy + decoding="async"; width/height em todas
Fontes locais preload + font-display: swap
Includes com caminhos absolutos baseados em [BASE_URL][BASE_PATH]
CWV ≥ 90 sem quebrar A11y

ANEXO 4
Vínculo de Entidade Externa via sameAs (Schema.org)

1. Propósito

Este anexo define o bloco `sameAs` a ser inserido dentro da entidade principal
(Person e/ou LocalBusiness) do JSON-LD já especificado na Seção 6.2. Seu objetivo
é declarar formalmente, para os mecanismos de busca, que os perfis e páginas
externas listadas abaixo pertencem à mesma entidade representada pelo Ecossistema
HLF — incluindo o catálogo eletrônico oficial hospedado e administrado por
terceiros (Herbalife), sobre o qual não há controle editorial direto.

2. Bloco JSON-LD a ser inserido

Este bloco deve ser adicionado como propriedade `sameAs` dentro do objeto
`@id: ".../HLF/#person"` (ou `#business`, conforme a entidade escolhida como
principal na Seção 6.2), mantendo-se a mesma estrutura de `@graph` já definida.

```json
"sameAs": [
  "https://catalogoherbalife.com.br/edusidegum",
  "https://instagram.com/edusidegum",
  "https://facebook.com/edusidegum",
  "https://twitter.com/edusidegum"
]
```

3. Exemplo de integração ao objeto Person existente (Seção 6.2)

```json
{
  "@type": "Person",
  "@id": "https://edusidegum.github.io/HLF/#person",
  "name": "Eduardo Sidegum",
  "sameAs": [
    "https://catalogoherbalife.com.br/edusidegum",
    "https://instagram.com/edusidegum",
    "https://facebook.com/edusidegum",
    "https://twitter.com/edusidegum"
  ]
}
```

4. Regras de manutenção

- Qualquer alteração de URL de rede social (troca de usuário, migração de

function openPostIt(index) {

5. PÁGINA DE REDIRECT COM SEO (INDEX.HTML)

Instrução: Toda página de redirect (meta refresh) DEVE conter no <head>:

<title>Windows 11 Maintenance Hub — Título Descritivo</title>
<meta name="description" content="Descrição com 150-160 caracteres para SEO.">
<meta name="robots" content="index, follow, max-snippet:-1, max-image-preview:large, max-video-preview:-1">
<meta name="googlebot" content="index, follow, max-snippet:-1, max-image-preview:large, max-video-preview:-1">
<meta name="bingbot" content="index, follow, max-snippet:-1, max-image-preview:large">
<link rel="canonical" href="https://edusidegum.github.io/PROJETO/index.html">
<link rel="alternate" hreflang="pt-BR" href="https://edusidegum.github.io/PROJETO/index.html">
<meta http-equiv="refresh" content="0;url=dashboard.html">

<meta property="og:title" content="...">
<meta property="og:description" content="...">
<meta property="og:image" content="https://edusidegum.github.io/PROJETO/assets/img/og-image.webp">
<meta property="og:url" content="...">
<meta property="og:locale" content="pt_BR">

JSON-LD obrigatório: Person (Eduardo Sidegum) + WebSite + WebPage + BreadcrumbList.

Funcionamento: O Googlebot lê as meta tags do <head> ANTES de seguir o redirect. Por isso o redirect instantâneo (content="0") não prejudica a indexação — desde que as tags estejam corretas.
